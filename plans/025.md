# Plan 025 — Fix verification failures: 011

## Problem Summary

Issue 011 (Rounding utilities) was marked COMPLETED but failed the `bun run check` (format + lint) verification gate. The **only failure** is a biome formatter violation in `tests/rounding.test.ts`. Build (`bun run build`) and tests (`bun test`) already pass.

## Root Cause

Biome's default line width is 80 characters. Several one-liner `it(...)` test cases in `tests/rounding.test.ts` exceed this limit and biome wants them wrapped. Additionally, biome prefers double quotes for strings containing escaped single quotes (the `describe('roundHalfEven (banker\'s rounding)')` line).

Specific lines that need reformatting:

1. **Line 7** — `it('rounds up when next digit >= 5', ...)` — too long, needs line break before `expect(...)`
2. **Line 10** — `it('handles floating-point edge case 1.005', ...)` — too long, needs line break
3. **Line 11** — `it('rounds to 3 decimal places', ...)` — too long, needs line break
4. **Line 19** — `it('returns same value if already exact', ...)` (ceil) — too long, needs line break
5. **Line 21** — `it('handles negative with decimals', ...)` (ceil) — too long, needs line break
6. **Line 27** — `it('floors to integer when decimals omitted', ...)` — too long, needs line break
7. **Line 28** — `it('floors to integer when decimals is 0', ...)` — too long, needs line break
8. **Line 30** — `it('returns same value if already exact', ...)` (floor) — too long, needs line break
9. **Line 32** — `it('handles negative with decimals', ...)` (floor) — too long, needs line break
10. **Line 37** — `describe('roundHalfEven (banker\'s rounding)')` — biome prefers double quotes when string contains escaped single quote: `describe("roundHalfEven (banker's rounding)")`

## Design Decision

**Approach: Run `biome format --write`** to auto-fix all formatting issues in one shot, rather than manually editing each line. This is the idiomatic approach — biome's formatter is deterministic and the `format` script is the source of truth for code style.

This is safe because:
- The changes are purely whitespace/quote-style (no semantic changes)
- Tests already pass and will continue to pass after formatting
- Build already passes (TypeScript type-check is unaffected by formatting)
