# Plan: formatCurrency utility

## Overview
Implement `formatCurrency(n: number, currency?: string): string` — a pure function that formats a number as a currency string (e.g. `1234.5` → `"$1,234.50"`). Internally reuses `formatWithCommas` (from issue 010-1) for thousand-separator formatting. Write comprehensive tests in `tests/formatting.test.ts`.

## Design Decisions

1. **Currency symbol lookup map** — maintain a small `Record<string, string>` mapping currency codes to symbols: `{ USD: '$', EUR: '€', GBP: '£', JPY: '¥' }`. Throw on unknown codes with a clear error message. This is intentionally minimal — the acceptance criteria only require these four currencies.
2. **Default to `"USD"`** — use nullish coalescing (`currency ?? 'USD'`) so callers can omit the parameter for the most common case.
3. **`toFixed(2)` for decimal formatting** — all supported currencies use 2 decimal places (even JPY, per the acceptance criteria's `"$1,234.50"` pattern). `toFixed(2)` handles rounding correctly.
4. **Reuse `formatWithCommas` internally** — after computing the absolute value's integer part, use `formatWithCommas` to add thousand separators to the integer part, then manually append the decimal part. This avoids duplicating comma-insertion logic. *Alternative considered*: call `formatWithCommas` on the full number, but that would preserve the original decimal places rather than forcing 2. Instead, we format manually: `toFixed(2)` → split on `.` → `formatWithCommas` on integer part → rejoin.
5. **NaN / ±Infinity guard-first** — return the literal string representations (`"NaN"`, `"Infinity"`, `"-Infinity"`) immediately, consistent with the `formatWithCommas` pattern. The acceptance criteria explicitly require these string literals (not throwing).
6. **Negative sign placement** — prepend `"-"` before the currency symbol: `"-$1,234.00"`. This matches the acceptance criteria example and the most common US convention.
7. **No `Intl.NumberFormat`** — keep the function pure and deterministic, consistent with the project convention of no side-effects/I/O and sibling formatting utilities.
8. **Shared test file** — append a `describe('formatCurrency', ...)` block to `tests/formatting.test.ts`. If the file doesn't exist yet (010-1 hasn't been built), create it; otherwise append.

```ts
// ── Currency formatting (issue 010-3) ───────────────────────────────────────

const CURRENCY_SYMBOLS: Record<string, string> = {
  USD: '$',
  EUR: '€',
  GBP: '£',
  JPY: '¥',
}

/**
 * Formats a number as a currency string (e.g. 1234.5 → "$1,234.50").
 * @param n - The number to format
 * @param currency - ISO 4217 currency code (default: "USD")
 * @throws Error if currency code is unknown
 */
export function formatCurrency(n: number, currency?: string): string {
  if (Number.isNaN(n)) return 'NaN'
  if (!Number.isFinite(n)) return n > 0 ? 'Infinity' : '-Infinity'

  const code = currency ?? 'USD'
  const symbol = CURRENCY_SYMBOLS[code]
  if (symbol === undefined) {
    throw new Error(`Unknown currency code: ${code}`)
  }

  const isNegative = n < 0
  const abs = Math.abs(n)
  const fixed = abs.toFixed(2)
  const [intPart, decPart] = fixed.split('.')
  const withCommas = formatWithCommas(Number(intPart))

  return (isNegative ? '-' : '') + symbol + withCommas + '.' + decPart
}
```

**Key implementation detail**: We call `formatWithCommas(Number(intPart))` to add thousand separators to the integer portion. Since `intPart` is always a non-negative integer string (from `Math.abs` + `toFixed` + `split`), this is safe and correct. `formatWithCommas` handles the comma insertion; we handle the currency symbol, sign, and fixed decimal places.

## Edge Cases & Risks

- **Dependency on 010-1**: `formatCurrency` calls `formatWithCommas` internally. Issue 010-1 must be built first (it is listed as a dependency in the issue). If 010-1 is not yet implemented, the build will fail at import/call time — this is correct and expected.
- **`-0`**: `Math.abs(-0)` → `0`, and `(-0 < 0)` is `false`, so `formatCurrency(-0)` → `"$0.00"`. This is acceptable (no spurious negative sign).
- **Rounding**: `toFixed(2)` uses banker's rounding in some edge cases, but for display purposes this is standard and acceptable.
- **Currency symbol map is not exported**: `CURRENCY_SYMBOLS` is a module-level constant, not exported. This keeps the public API clean. If future issues need to extend supported currencies, the map can be exported or made configurable then.
- **JPY decimal places**: In real-world usage, JPY typically has 0 decimal places. However, the acceptance criteria show a uniform 2-decimal format, so we use `toFixed(2)` for all currencies. This can be revisited in a future issue if needed.
